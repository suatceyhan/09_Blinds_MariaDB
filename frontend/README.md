# Frontend (React + Vite)

Modern panel: **Tailwind CSS v4**, **Plus Jakarta Sans**, giriş ekranında split layout; giriş sonrası üst bar + kenar çubuğu (`AppLayout`). **Desktop (lg+):** sidebar header’da **Collapse / Expand menu** (`PanelLeftClose` / `PanelLeft`); dar modda yalnızca ikonlar + `title` ipuçları; alt grup tıklanınca ilgili hub’a gider. Tercih **`localStorage`** anahtarı `blinds_sidebar_nav_collapsed` (`1` = dar). Mobil: tam genişlik drawer + üst **Menu** düğmesi (daraltma yalnızca geniş ekranda).

## Geliştirme

```bash
cd frontend
npm install
copy .env.example .env   # isteğe bağlı; varsayılan VITE_API_BASE=/api
```

Aynı makinede FastAPI’yi açın. Bu repoda Vite, **`09_Blinds` ile yan yana çalışmak** için API’yi varsayılan olarak **8810** portuna proxy’ler (`vite.config.ts`; `VITE_DEV_API_ORIGIN` ile değiştirilebilir). Tek başına veya paralel çalıştırırken:

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8810
```

```bash
npm run dev
```

Tarayıcı: [http://127.0.0.1:5173](http://127.0.0.1:5173)

**İki kopyayı aynı anda:** `09_Blinds` → API **8000**, UI **5174**. Bu repo → API **8810**, UI **5173**. İki ayrı backend terminali + iki ayrı frontend terminali yeterli. (`8001` zaten kullanılıyorsa veya Windows izin hatası veriyorsa `8810` kullanın; proxy adresini `.env` içinde **`VITE_DEV_API_ORIGIN`** ile eşleştirin.)

## Auth

- `POST /auth/login` gövdesi **x-www-form-urlencoded** (`email`, `password`).
- `POST /auth/register` JSON ile kayıt; token’lar `starter_app_access_token`, `starter_app_refresh_token` (**localStorage**, tarayıcı kapanınca bile kalabilir).
- **Hareketsizlik:** Varsayılan **30 dk** içinde fare/klavye/API etkileşimi yoksa oturum temizlenir ve `/login` açılır. `.env`: `VITE_IDLE_LOGOUT_MINUTES` (dakika); `0` ile kapatılır. Süre, `starter_app_last_activity_ms` ile saklanır; PC yeniden açıldıktan sonra bu süre aşıldıysa doğrudan giriş ekranı görünür.

## UI conventions (this template)

- **English** end-user copy: menus (`appPages.ts`), auth, dashboard, settings, reports, layout.
- **Addresses:** single-line fields use **`AddressAutocompleteInput`** (`src/components/ui/AddressAutocompleteInput.tsx`): after **3+ characters**, debounced suggestions from **Photon / OpenStreetMap** (`src/lib/photonAddressSuggest.ts`, public `photon.komoot.io` — needs browser network access). Post-filtering matches **tokens** in the suggestion line (partial street typing such as `fred mc` still matches `Frederick …`), not a strict whole-line prefix. **Settings → Company info** (and superadmin **Companies**) set **`country_code`** (`countrycode=` on Photon) and optional **`region_code`** (Canadian provinces/territories or US states + DC from `src/lib/companyRegions.ts`). When region is set, Photon gets **lat/lon/zoom** bias, the search string gains **province/state context** (e.g. `…, Ontario, Canada`), and only hits in that subdivision are shown when Photon tags them (`state` / `statecode` or the formatted line contains the province name). If the user types a **leading house number** and Photon returns street segments without it, the UI **prepends** that number for a readable civic line. **Autocomplete suggestion lines omit postal/ZIP** (entered in the separate Postal code field where the form provides one). `/auth/me` includes **`active_company_country_code`** and **`active_company_region_code`** for customer/estimate address fields. Company country dropdown is **CA / US / Any** plus a **legacy** option if the stored country is outside that list. Hint text under address fields (no sample placeholder in the input); **lists and detail views** use **`AddressMapLink`** → **Google Maps** (`src/lib/googleMaps.ts`). DB: **`DB/23_companies_country_code.sql`**, **`DB/24_companies_region_code.sql`**. Not Canada Post / AddressComplete.
  - **Postal code is a separate, optional field** (manual) because OpenStreetMap/Photon postcodes can be street-segment approximations (not door-specific).
- **Confirmations:** centered `ConfirmModal` (`src/components/ui/ConfirmModal.tsx`) for discard / remove / deactivate flows — not `window.confirm`. Optional **`secondaryAction`** adds a third button between cancel and primary confirm.
- **Soft-deleted rows:** Roles, User roles, **Estimates** ve **Orders** listelerinde **Show deleted** (`ShowDeletedToggle`); estimates için **Restore** (`POST /estimates/:id/restore`, `include_deleted=true` list parametresi). Orders için: `DELETE /orders/{id}` soft delete, listede `GET /orders?include_deleted=true`, restore: `POST /orders/{id}/restore`. Bootstrap superadmin assignment is protected: it shows **Protected** (not **Remove**) and the role dropdown does not allow selecting `superadmin`.
- **Kayıt (DWP TM hizası):** `/register` her zaman **Employee** ve **Company** kartlarını gösterir. Sunucu anında kayda izin veriyorsa (`PUBLIC_REGISTRATION_ENABLED=true`) üçüncü kart **Instant signup** → `/register/direct`. Başvurular `/verify-email?token=…&type=employee|company`. Superadmin **Settings → Pending applications**.
- API error `detail` strings may still be non-English if the backend returns them that way.

## Yapı

- `src/app` — router, layout
- `src/features` — sayfalar (auth, dashboard, customers, **estimates**). **Dashboard** (`/dashboard/summary`) refetches when **`active_company_id`** changes (company switch in the header).
- `src/lib` — `api.ts`, `authStorage.ts`, `sessionIdle.ts`, `useSessionIdleTimeout.ts`
- `src/components/ui` — paylaşılan modal vb.
- **Schedule:** `/schedule` — view-only calendar for **Estimates** visits and **Orders** installations (Day/Week/Month/Schedule views). In Month view, days outside the current month are visually muted and not interactive.
- **List row actions** (Customers, Estimates, Companies, Users, Orders): icon-only **View / Edit / delete / Restore** share the same bordered control pattern as **Orders**; workflow text buttons (e.g. **Mark pending**, **Make order**, **Next: …**) stay separate. Workspace rule: **`.cursor/rules/blinds-list-table-actions.mdc`**.
- **Settings → Contract / Invoice:** printable HTML documents (Deposit invoice + contract, Final invoice) are generated from DB and opened in a new tab for browser Print to PDF. Permissions: `settings.contract_invoice.view|edit`.

Yeni modüller: `src/features` altına ek sayfalar ve `AppRoutes` + `appPages.ts` menüsünü genişletin.

**Estimates:** `/estimates` — search, schedule filter, **status filter** chips from **`GET /estimates/lookup/estimate-statuses`**; new-estimate **prospect address** uses **`AddressAutocompleteInput`** (same Photon hints as other address fields). (same **`sort_order`** as Lookups; filter query **`status_esti_id`**); built-in **`code`** values keep fixed chip colors, custom names use the same **name-based** palette as the orders list. **Show deleted**, **Status** column (label from **`status_estimate`**); **Visit date** column. **Actions**: **Mark pending** (text) when **`status`** is **`new`** — `PATCH /estimates/:id` with matrix **Pending** id from **`GET /estimates/lookup/estimate-statuses`**. **Make order** (text link) when **`pending`** and **`orders.edit`**, plus edit/remove icons, or view + restore when deleted. **`/estimates/:id`** view (same **Make order** rule), **`/estimates/:id/edit`** PATCH form (**`status_esti_id`** from lookups when status is not **Converted**). **Converted** rows: status is read-only on edit (link to Orders); **`PATCH`** omits **`status_esti_id`**; cancelling the linked order (soft-delete or order status whose name contains *cancel*) sets the estimate **Cancelled** on the server; restore / move off cancel sets estimate back to **Converted** if it was **Cancelled**. New estimate: compact modal — **Prospect** (name/contact; not stored under Customers until an order is saved) or **existing customer**; per blinds type optional **line amount**. **Visit date and time**: **VisitStartQuarterPicker** — date + time triggers open a **calendar + 15-minute time list** (fixed popover); date trigger **`Apr 1, 2026`**, time **`2:45 PM`** → `scheduled_wall`; estimates list **Visit date** column uses **`Apr 1, 2026 - 2:30 PM`**. **Time zone** beside it. **Organizer & employees**: organizer row locked on, optional employee checkboxes. **Blinds types**: same grid as **New order** (**Qty**, **Category**, **Amt**, **Line note**) via **`GET /estimates/lookup/blinds-order-options`**; line notes persist on **`estimate_blinds`** and pre-fill **`orders.blinds_lines`** when converting. List **Types & windows** shows category and note snippets when set. **`DELETE /estimates/:id`** soft delete.

**Orders:** `/orders` — list table: **Customer**, **Status**, **Agreement date**, **Day Past** (calendar days since agreement), **Installation date** (`installation_scheduled_start_at`), **Total** (subtotal + tax), **Paid** (down + `final_payment`), **Balance**, **Actions** (no **Created** column on the list; detail still shows dates where relevant). **Agreement date** on the create form defaults to **today** (local date). **Make order** (`/orders?fromEstimate=…`): prefill aligns **category** with the blinds matrix when the estimate payload omits it; estimate **prospect** may leave Customer empty until save (server creates the customer) — **Create order** stays enabled whenever the estimate is linked, matching **`POST /orders`** (optional **`customer_id`** when **`estimate_id`** is sent). **Line amount** zero from the API shows as an empty field (placeholder) so you can type without clearing a stray `0`. **Edit order**: **Ready for installation** allows optional **installation** date/time (not required to save); when set and Google Calendar is connected, the backend syncs an installation event. **New / view / edit**: **Attachments** — **Take photo** (camera on supported devices), **Upload photo**, **Upload Excel**; on create, queued files upload after the order is created. **Recorded payments** (view): remove a **Pay** line (not down payment) via trash + **ConfirmModal**; **`DELETE /orders/.../payment-entries/...`**. **Show deleted** includes inactive (`active` false) rows; deleted rows are styled and omit edit; **Restore** opens **ConfirmModal** before `POST /orders/{id}/restore`. **New order** and **Edit order** use **`GET /orders/lookup/blinds-order-options`**: blinds grid is a **table** — **one row per blinds type** (checkbox + name), **columns** = **Qty** (number input 1–99 with stepper arrows), **Category** (column width follows header + longest option label; full value in tooltip / dropdown), **line amount** (up to six digits before the decimal, two after), **line note** last. If the table is wider than the card, **scroll horizontally inside the blinds section** so line notes stay reachable; the order modal body does not use a horizontal scrollbar. **Order note** sits full-width on the form grid like **Customer** / **Status**. The financial block shows **Total (incl. tax)** (line subtotal + tax), **Down payment**, **Taxable base**, then **Paid** (down + server **`final_payment`** on edit/view; create shows down only), **Balance due**, **Tax**. The API keeps **`orders.total_amount`** as the line subtotal and recalculates it from **`blinds_lines`** on create/patch. **Order detail**: **Installation** (single date and time) on **view** and **edit** (not on **New order**) uses **VisitStartQuarterPicker** like estimates; editable only when status matches **Ready for installation** (name substring) on an active order, otherwise read-only; **`PATCH`** clears **`installation_scheduled_end_at`**. **Payment** opens a modal to record an amount; **`POST /orders/{id}/record-payment`** appends a row in **`order_payment_entries`** and updates **`final_payment`** / **balance**. Between the financial block and **Dates**, **Recorded payments** lists down payment (if any) and each **Pay** amount with date/time, chronological order. **Order note** (full order) plus per-line notes. **Status** from **`GET /orders/lookup/order-statuses`**, agreement date; list and view use **badges** with the same colors as the status **filter** chips (name-based: new / production / ready+install / done / …). **Actions**: advance shows **`workflow_next_status_label`** from **`GET /orders`** (first workflow transition target; fallback **Next**) / **Balance review** — advance opens **ConfirmModal** before **`PATCH`** status advance (when advancing to **Ready for installation**, copy explains optional installation and adds **Enter installation now** → status **`PATCH`** then opens **order view**); **Balance review** (same compact style as estimate **Mark pending** / **Make order**), placed at the **start** of the action group (toward the data columns); then view, edit, delete/restore. Workflow matches (**New**→**In production**→**Ready for installation** by substring on matrix status names); from **Ready for installation**, **Balance review** opens **order view** and an **information** modal with **balance due** (set **Done** in **Edit** when ready). Rows with **Done** status or **zero balance** use the same light **emerald** tint; the **API** keeps them aligned (fully paid ⇒ **Done** when a matrix status name contains “done”; **Done** with a positive balance moves to **Ready for installation** when that status is on the company matrix, otherwise the first non-done / non-cancel matrix status on save; **PATCH** rejects leaving **Done** while still fully paid, or choosing **Done** while balance remains). **Edit**, delete/restore. Company rate: **Settings → Company info**. **Deep links:** **`/orders?viewOrder=<id>`** opens the same order detail overlay (list **View** sets this query so refresh matches). **Customer & links** includes **Edit** (with **`customers.edit`**) → **`/customers?edit=<id>`** opens the customers list edit modal; estimate **Edit** (with **`estimates.edit`**) goes to **`/estimates/<id>/edit`** unless the linked estimate is **converted** (then view-only copy; the estimate link still opens the read-only view). **Completed paid jobs (`job_edit_locked` on **`GET /orders/{id}`**):** edit disables lines, payments (including deleting recorded payments), additional orders, attachments/line photos, and **Save**; **Add expense** stays enabled — use **New order** for more products.

**Customers / Companies (list):** **`/customers?edit=<customerId>`** (with **`customers.edit`**) opens the edit modal and clears the query; **`/companies?edit=<companyId>`** does the same for superadmin company edit. **Customer view** / **Company view** profile pages include **Edit** when allowed (customer: **`customers.edit`**; company: superadmin).

**Sidebar accordions** (Lookups, Reports, Settings, Permissions): one row (chevron + icon + label). **First click** opens the subtree and navigates to that group’s hub (`/lookups`, `/reports`, `/settings`, `/permissions`). **Second click** (while open) collapses the accordion without changing route until you pick a child link.

**Lookups:** `/lookups` — static overview. **`/lookups/blinds-types`** and **`/lookups/blinds-product-categories`** use the same section layout as order statuses (violet header card, add block, manage table, company×column matrix with **`Save matrix`**). Blinds types API: **`/permissions/blinds-type-matrix`**; product categories: **`/permissions/product-category-matrix`**. List endpoints support **`catalog_scope=tenant`** (default) vs **`global`** for the manage table when the user has edit permission. **`/lookups/estimate-statuses`** and **`/lookups/order-statuses`** use **`/permissions/estimate-status-matrix`** and **`/permissions/order-status-matrix`**. **Role / User permissions matrix:** Lookups shows the **full submenu tree**; each child row has its own **`.view` / `.edit`** keys (e.g. **`lookups.blinds_types.view`**; status grids use **`settings.estimate_status_matrix.*`** / **`settings.order_status_matrix.*`**). The **Lookups** hub row keeps **`lookups.view` / `lookups.edit`**. API routes accept the **granular** key **or** the legacy broad key.

**Reports:** `/reports` hub has **no** separate “Overview” row in the sidebar or **Role permissions** matrix (same URL and keys as the parent Reports row). Sub-nav: **Operational** → **Quarterly summary** → **Detail view** (toolbar demo). Hub pages are text-only; use the sidebar to drill in.

**Reports → Financial:** `/reports/financial` — monetary reports with a modern date range filter (presets + custom):
Revenue (incl. tax), Collected (down payment + recorded payments), A/R balance, Profit (revenue − expenses), Tax,
Taxable base, and a Revenue vs Collected trend table (daily/weekly).

**Settings hub:** `/settings` — static overview; sidebar items use **scoped** keys (not the main **Companies** menu): **Company info** (`settings.company_info.*`), **Integrations** (`settings.integrations.*`), **Blinds line matrices** (`settings.blinds_line_matrices.*`), **Pending applications** (`settings.pending_applications.*`). The **Settings** row itself remains **`settings.access.*`** (hub only).

**Settings → Blinds line matrices:** **`/settings/blinds-line-matrices`** — **product category** × blinds type matrix (rows = categories, columns = types). **Save all changes**. Legacy **`/settings/blinds-category-matrix`** and **`/settings/blinds-extra-matrix/:kindId`** redirect here (extra-attribute matrices were removed).

**Permissions hub:** `/permissions` — static overview; **Permissions** ana satırı **`permissions.access.*`** ( **`settings.access.*` ile karışmaz** ). Alt sayfalar: **Roles**, **Role permissions**, **User roles**, **User permissions**. Estimate/order status grids and product categories (with matrix) live under **Lookups**. Legacy `/settings/roles` **redirect** `/permissions/…`. **Role permissions (matrix):** satır `page id` + ortak anahtar düzeltmeleri `roleMatrixTreeLogic.ts` içinde.

**Settings → Company info:** `/settings/company-info` — active company (header switcher) için ad, iletişim, adres ve **default sales tax (%)**; siparişte **taxable base** × bu oran = **tax amount** (sunucuda `orders.tax_amount`). Kayıt **`PATCH /companies/{id}`** — UI izinleri **`settings.company_info.*`**; API ayrıca eski roller için **`companies.edit`** / **`companies.view`** ile **OR** kabul eder.

- **Company logo**: upload/remove company logo (PNG/JPG/WebP/GIF, max 2MB) from the same page. Backend stores files under `UPLOAD_ROOT` (default `backend/data/uploads`) and serves them at `/uploads/...` (Vite dev proxy forwards `/uploads` to FastAPI).

**Integrations:** `/settings/integrations` — Google Calendar OAuth (**`settings.integrations.*`**; API’de `companies.*` ile OR); yeni estimate’ler bağlı takvimde etkinlik oluşturabilir (backend `.env` + `docs/GOOGLE_CALENDAR_SETUP.md`).
