# Database scripts

- **`blinds-postgresql-reference.sql`**: Reference only ŌĆö same PostgreSQL schema as `09_Blinds/DB/blinds-postgresql.sql` (used while comparing behaviour during the MariaDB port).

- **`blinds-mariadb.sql`**: Generated by **`scripts/generate_mariadb_sql.py`** from the reference file. Intended for a **fresh MariaDB 10.11+** database (UTF-8); **12.2** is supported (same feature set as 10.11+ for this script). Row-level security and PostgreSQL-only constructs are removed or rewritten; tenant isolation is enforced in application code (`TENANT_RLS_ENABLED=false`).

```bash
python scripts/generate_mariadb_sql.py
mysql -u USER -p -e "CREATE DATABASE your_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
mysql -u USER -p your_db < DB/blinds-mariadb.sql
```

**Notes**

- Use **`DEFAULT (UUID())`** expectations (MariaDB 10.7+). Primary keys use **`CHAR(36)`**.
- Bare PostgreSQL **`VARCHAR`** (without `(n)`) is emitted as **`VARCHAR(512)`** in the generated file ŌĆö MariaDB requires an explicit length.
- PostgreSQL **partial unique indexes on expressions** (e.g. `lower(email)` + `WHERE ŌĆ”`) are converted to **`GENERATED ALWAYS ŌĆ” STORED`** helper columns plus a plain **`CREATE UNIQUE INDEX`** ŌĆö MariaDB does not support that PostgreSQL syntax directly. Those helper columns are added with **`ADD COLUMN IF NOT EXISTS`** so re-importing into an existing database does not fail with duplicate column names when **`CREATE TABLE IF NOT EXISTS`** is skipped.
- **Composite tenant FKs**: tables that reference **`companies(id)`** on **`company_id`** and also define **`FOREIGN KEY (company_id, ŌĆ”)`** to other tables use an explicit **`fk_<table>_company`** constraint on **`company_id`** (column-level `REFERENCES companies(...)` alone caused InnoDB errno 150 on CREATE TABLE). **`customers`** also declares **`KEY idx_customers_company_status_user (company_id, status_user_id)`** because the composite FK does not share the second column of **`PRIMARY KEY (company_id, id)`** ŌĆö InnoDB requires the FK columns as the leading index prefix.
- **Composite FK + `ON DELETE SET NULL` (PostgreSQL) vs InnoDB**: PostgreSQL allows **`ON DELETE SET NULL`** on a composite FK when only the nullable column(s) are cleared. InnoDB/MariaDB can fail **`CREATE TABLE`** (errno 150) for the same DDL. The generated file uses **`ON DELETE RESTRICT`** on **`fk_customers_status_user`** and **`fk_orders_status_order`**, and adds **`BEFORE DELETE`** triggers **`tr_status_user_before_delete_clear_customers`** and **`tr_status_order_before_delete_clear_orders`** to null out **`customers.status_user_id`** / **`orders.status_orde_id`** first so deletes on **`status_user`** / **`status_order`** still behave like PostgreSQL.
- The dump starts with **`SET NAMES utf8mb4 COLLATE utf8mb4_unicode_ci`** so UUID **`CHAR(36)`** FK pairs match across tables from the same connection defaults.
- Re-running the dump: each **`ALTER TABLE ŌĆ” ADD CONSTRAINT ŌĆ” FOREIGN KEY`** is preceded by **`DROP FOREIGN KEY IF EXISTS`** so you can re-import more safely if a previous run partially applied (prefer a **fresh database** for a clean install).
- Migration **32** (`global blinds_type`): the PostgreSQL `jsonb` rewrite of `orders.blinds_lines` is **not** reproduced in MariaDB SQL; use a custom script if you migrate legacy JSON data from PostgreSQL.
- If **`permissions`** inserts fail**, ensure the MariaDB reserved word **`key`** is quoted as shown in the generated file (the generator handles this).
